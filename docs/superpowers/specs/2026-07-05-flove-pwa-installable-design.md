# flove as an installable PWA + root cleanup — design

**Date:** 2026-07-05
**Status:** approved (design), pending implementation plan

## Problem

People who get flove are confused about *which file to click* to start it. The
downloaded folder mixes the entry point with scripts, license and readme, and on
mobile there is no way to have flove behave like "another app in your list."

We want flove to install like a normal app — one icon on mobile and desktop,
fullscreen, working offline with its files stored on the device — and we want the
root folder to be as clean as possible.

## Solution in one line

Make flove a **PWA** (manifest + app icons + full-offline service worker) so it
installs straight from flove.org, and **declutter the served root** by moving the
icon into `images/` and README/LICENSE into `docs/`.

## Decisions (from brainstorming)

| Question | Decision |
|---|---|
| Direction | **PWA-first**: installable from flove.org; the zip stays as a secondary offline copy |
| Offline scope | **Full precache** (~5 MB): every app works offline immediately after install |
| Icon opens to (`start_url`) | **`launch.html`** (EN/ES gate), which is **repointed to forward to `index.html`** (home) instead of `apps/appy/appy-mini.html` |
| README / LICENSE | **Hard move both** into `docs/` — nothing left at root |
| Scope of this pass | Full design → spec → plan → build everything (cleanup + manifest + icons + service worker + repoint) |

## Non-goals

- No app-store packaging (APK/IPA). PWA install only.
- No change to the hosting/publish pipeline (still Gitea → `update-web` → flove.org).
- The discreet in-app "Install flove" button (`beforeinstallprompt`, §13.6 onboarding
  style) is **deferred** to a follow-up; this pass relies on the browser's own
  install affordance.
- The Linux `.desktop` + `start-flove.sh` local-server fallback is **kept as-is**
  (Firefox desktop can't install PWAs); only its icon path is updated.

## Key context discovered

- `start-flove.sh` and `flove-localhost.desktop` are **not tracked** — they are
  git-ignored and **generated by `build-flove-zip.sh`**. The copies at the working
  root are machine-specific local artifacts. So the *served/tracked* root only
  needs `flove-icon.svg`, `README.md`, `LICENSE` relocated.
- `flove.zip` is built from `git archive HEAD` (the committed site), so any moved
  file must be committed before the zip is rebuilt, and generated-file references
  (the `.desktop` `Icon=`) must be updated in `build-flove-zip.sh`.
- flove.org is GitHub Pages over HTTPS — satisfies the PWA secure-context and
  service-worker-at-root requirements. `sw.js` served from `/` controls the whole
  scope with no extra headers.

## Component design

### A. Root cleanup + reference updates

Move:
- `flove-icon.svg` → `images/flove-icon.svg`
- `README.md` → `docs/README.md`
- `LICENSE` → `docs/LICENSE`

Update every reference:

| File | Line | Current | New |
|---|---|---|---|
| `launch.html` | 53 | `src="flove-icon.svg"` | `src="images/flove-icon.svg"` |
| `build-flove-zip.sh` | 85 | `Icon=flove-icon.svg` (generated `.desktop`) | `Icon=images/flove-icon.svg` |
| `docs/README.md` | — | tree diagram lists `flove-icon.svg` at root | update paths to reflect new layout |
| `publish-lowai.sh` | 25 | `--exclude='README.md'` | harmless once README moves; leave or update comment |

Notes:
- `docs/index.html:4036` links to a Cloudinary raw copy of `README.md` — external
  URL, unrelated to repo path; no change needed.
- After moving, the Gitea/GitHub repo page shows no readme card or auto-detected
  license (accepted trade-off).

### B. `manifest.webmanifest` (root)

```json
{
  "name": "flove",
  "short_name": "flove",
  "start_url": "/launch.html",
  "scope": "/",
  "display": "standalone",
  "background_color": "#fffdfb",
  "theme_color": "#ff8a3a",
  "icons": [
    { "src": "/images/icons/flove-192.png", "sizes": "192x192", "type": "image/png", "purpose": "any" },
    { "src": "/images/icons/flove-512.png", "sizes": "512x512", "type": "image/png", "purpose": "any" },
    { "src": "/images/icons/flove-512-maskable.png", "sizes": "512x512", "type": "image/png", "purpose": "maskable" }
  ]
}
```

- `background_color`/`theme_color` taken from the existing palette (`--paper`,
  gradient mid stop). A manifest theme color must be a single color, not the
  gradient.
- `display: standalone` (fullscreen chrome-less window with OS status bar). Can be
  revisited to `fullscreen` later.

### C. App icons (`images/icons/`)

Generated from `images/flove-icon.svg` (a clean gradient + 3 paths, no clip/filter/`<use>`):

- `flove-192.png`, `flove-512.png` — transparent background, `purpose: any`.
- `flove-512-maskable.png` — art padded to ~80% safe zone on an **opaque** paper
  background so Android's circular/rounded mask never clips it. Produced from a
  small wrapper SVG (background rect + scaled-down art).
- `apple-touch-icon.png` (180×180, opaque) at root for iOS home-screen.

Rasterize with **headless Chrome** (`google-chrome --headless --screenshot`), not
ImageMagick `convert` (unreliable on SVG gradients). **Read each PNG to verify**
appearance before committing (rasterize-to-verify rule).

### D. Service worker `sw.js` (root) + build step

- New `build-sw.mjs`: enumerates the tracked, servable site files (same source of
  truth as the zip — `git ls-files`, filtered to what the site serves), writes a
  `PRECACHE` array and a `VERSION` (content hash) into `sw.js`.
- `sw.js` behavior:
  - `install`: `caches.open(VERSION)` → `addAll(PRECACHE)` → `skipWaiting()`.
  - `activate`: delete caches whose name ≠ `VERSION`; `clients.claim()`.
  - `fetch`: cache-first, fall back to network (and network-fill on miss).
- Wire `build-sw.mjs` into `build-flove-zip.sh` (or run before publish) so the
  precache list/version always matches the deployed site.
- Precache list should include `launch.html`, `index.html`, `apps/**`, `images/**`,
  `docs/flove.js` and other runtime assets; exclude dev-only files already excluded
  from the zip.

### E. Entry-page wiring

On `launch.html`, `index.html`, `apps/index.html` add in `<head>`:

```html
<link rel="manifest" href="/manifest.webmanifest">
<meta name="theme-color" content="#ff8a3a">
<link rel="apple-touch-icon" href="/apple-touch-icon.png">
<meta name="apple-mobile-web-app-capable" content="yes">
<meta name="apple-mobile-web-app-title" content="flove">
```

And a small registration script (inline or shared `docs/flove.js` helper):

```js
if ('serviceWorker' in navigator) {
  window.addEventListener('load', () =>
    navigator.serviceWorker.register('/sw.js').catch(() => {}));
}
```

### F. `launch.html` repoint

Change `ENTRY` (currently `apps/appy/appy-mini.html`) → `index.html`. This changes
the live launch flow for everyone (gate → home), not only the installed app.

## Publish & verify

1. Commit moves + edits to Gitea (`marc/flove`, main).
2. Regenerate icons, `sw.js` (via `build-sw.mjs`), and `flove.zip`.
3. `update-web` → floveorg.github.io → flove.org.
4. Verify on flove.org:
   - manifest loads, no console errors, "Install" offered (Chrome/Android).
   - service worker registers; DevTools → Application shows precache populated.
   - offline toggle: apps still load.
   - installed icon opens the gate then home; icon renders correctly (incl.
     maskable on Android).
   - iOS: Share → Add to Home Screen works, correct touch icon.
